# Decision: 서브에이전트 무한루프 방지를 위한 하이브리드 차단 도입

## 결정

말단 워커(서브에이전트)가 다른 서브에이전트를 재귀적으로 호출하며 무한 루프에 빠지는 현상을 방지하기 위해 **프롬프트 레벨 + 플러그인 레벨의 이중 방어 계층**을 도입한다.

**첫 번째 계층 (프롬프트 레벨):**

- `experimental.chat.messages.transform` 훅에서 현재 에이전트가 서브에이전트일 경우 `subagent-turn-end.md` 템플릿을 주입
- 템플릿에 "네트워크 자원 보호" 및 "호출 권한 회수"를 명시하여 재귀 호출을 언어적 차단

**두 번째 계층 (플러그인 레벨):**

- `tool.execute.before` 훅에서 정규식 기반 패턴 블록 적용
- 차단 패턴: `^task$`, `^background_`, `agent` (오케스트레이션 도구 식별)
- 서브에이전트가 오케스트레이션 도구를 시도할 경우 하드 에러 발생

## 근거

단일 방어선은 우회 가능성이 있다. 프롬프트만으로는 에이전트가 지시를 무시할 수 있고, 훅만으로는 새로운 도구나 패턴 변형에 취약하다.

- **이중 방어**: 언어적 차단(soft) + 기술적 차단(hard)으로 우회 난이도를 기하급수적으로 증가
- **명확한 책임 분리**: 프롬프트는 "왜"(의도), 훅은 "어떻게"(강제)를 담당
- **네트워크 자원 보호**: 재귀 호출은 exponential한 API 호출을 유발하며 비용 폭증 가능

## 고려한 대안

- **프롬프트 차단만 사용**: 탈락 — LLM이 지시를 따르지 않거나 jailbreak당할 수 있음
- **훅 차단만 사용**: 탈락 — 새로운 오케스트레이션 도구 추가 시 차단 패턴 업데이트 필요, false positive 가능성
- **서브에이전트에 플래그 전달**: 탈락 — 플래그 조작 가능성, 복잡한 상태 관리 필요
